# Plot Mod - 材质组合任务书：主材质 + 点缀材质按比例混合

> 面向 Cursor 的开发任务书。目标：道路路面材质、建筑外墙材质、建筑楼板材质这三处，从"单一材质"升级成"主材质 + 可选点缀材质，按比例随机混合"，让大面积表面看起来不那么单调。

---

## 0. 两个必须遵守的设计前提

1. **混合结果必须是确定性的，不能是真随机**。同一个位置（同一个 `BlockPos`）在同一条路/同一栋楼上，不管重新计算预览多少次，选出来的材质必须是同一个——否则用户没法稳定判断"这个比例看起来对不对"，而且预览和最终落地的结果会对不上。做法：用方块坐标（加上这条边/这栋建筑的 ID 做区分，避免不同道路在同一坐标上选出一样的图案）算出一个确定性的伪随机数，不要用 `Math.random()`/`ThreadLocalRandom` 这种每次结果不同的随机源。
2. **这次范围严格限定为三处**：道路路面材质（`RoadCrossSection.carriagewayMaterial`）、建筑外墙材质（`BuildingFootprint.wallMaterial`）、建筑楼板材质（`BuildingFootprint.floorMaterial`）。人行道、路肩、地基、屋顶、门窗这些材质这次**不要**顺带改，保持单一材质不变，范围以后再扩展。

---

## 1. 数据模型：`MaterialMix`

新建 `com.plot.core.material.MaterialMix`（放在核心层，不属于任何具体插件，供道路/建筑插件共用，参考之前把 `PolygonRegionUtils`/`PluginUiColors` 放到公共层的做法）：
```java
public class MaterialMix {
    private String primaryMaterial;
    private String accentMaterial;   // 可以为 null，表示不启用点缀，等价于纯单一材质
    private float accentRatio;       // 0.0-0.5，点缀材质占比，setter 做范围校验

    // 兼容旧数据：提供一个从纯字符串构造的静态方法
    public static MaterialMix single(String materialId) {
        return new MaterialMix(materialId, null, 0f);
    }

    public boolean hasAccent() {
        return accentMaterial != null && !accentMaterial.isBlank() && accentRatio > 0f;
    }
    // getter/setter，toJson/fromJson 或者作为普通 POJO 被 Gson 直接序列化都可以，参考现有 RoadEdge 字段的序列化方式
}
```

**改动范围**：`RoadCrossSection.carriagewayMaterial`、`BuildingFootprint.wallMaterial`、`BuildingFootprint.floorMaterial` 这三个字段，类型从 `String` 改成 `MaterialMix`。**这是一个会影响现有存档格式的改动**——`RoadNetwork`/`BuildingProject` 的 JSON 序列化里，这三个字段原来是纯字符串，现在要变成对象。为了不破坏旧存档，读取的时候要做兼容：如果读到的是旧格式的纯字符串（JSON 里这个字段直接是个 string 而不是 object），自动转换成 `MaterialMix.single(旧字符串)`；写的时候统一按新格式（对象）写出。这个兼容逻辑要写测试专门验证，不要假设以后不会有人拿旧存档打开。

## 2. 确定性加权混合：`MaterialMixResolver`

新建 `com.plot.core.material.MaterialMixResolver`（纯函数，不依赖 World）：
```java
public static String resolve(MaterialMix mix, BlockPos pos, String seedKey);
```
算法：
1. 如果 `!mix.hasAccent()`，直接返回 `mix.getPrimaryMaterial()`，不做任何随机计算（性能优化，绝大多数材质字段不会启用点缀，避免每个方块都算一次哈希）；
2. 否则，用 `pos.getX()/getY()/getZ()` 和 `seedKey`（传入这条边或这栋建筑的 ID 字符串）一起算一个确定性哈希（可以用简单的位运算组合几个坐标值和字符串的 `hashCode()`，不需要密码学级别的哈希，只要"同样的输入永远同样的输出、不同输入看起来分布均匀"就够），映射到 `[0, 1)` 区间的一个浮点数；
3. 这个浮点数小于 `mix.getAccentRatio()` 就返回点缀材质，否则返回主材质。

配单元测试：同一个 `pos`/`seedKey` 多次调用结果一致（确定性）；固定 `mix`，对一批不同 `pos` 统计返回点缀材质的比例，应该大致接近设定的 `accentRatio`（允许一定误差范围，不要求精确相等，毕竟是伪随机分布）；`seedKey` 不同时，同一个 `pos` 应该有不同的选择结果概率分布（避免两条不同的路在同一个世界坐标上出现一模一样的花纹图案）。

## 3. 接入生成逻辑

### 3.1 道路

`RoadGenerator.generateCarriagewayBlocks(...)` 现在接收的是一个已经解析好的固定 `blockId` 字符串参数，这次要改成接收 `MaterialMix`（连同一个 `seedKey`，用这条边的 `edgeId` 就行），内部原来"对每个方块位置写入 `blockId`"的地方，改成先调用 `MaterialMixResolver.resolve(mix, pos, edgeId)` 算出这个具体位置该用哪个材质，再写入。调用方那一行：
```java
generateCarriagewayBlocks(
    solids, metrics, segments, heightInfos, bridges, tunnels, terrain,
    crossSection.carriagewayWidth,
    crossSection.carriagewayMaterial,   // 现在是 MaterialMix，不再是提前解析好的字符串
    unitsPerBlock);
```
注意 `crossSection.carriagewayMaterial` 里存的 `primaryMaterial`/`accentMaterial` 目前可能是材质 key（比如 `material.plot.gravel` 这种，需要走 `getBlockIdFromMaterial` 转成真实方块 ID）也可能已经是真实方块 ID，具体看现有 `RoadCrossSection` 里这个字段之前存的是哪种格式，`MaterialMixResolver.resolve` 返回的应该是**已经转换过、可以直接用于放置的真实方块 ID**，如果传入的是材质 key，需要在 `resolve` 内部或者调用前先转换好，不要把这个转换逻辑漏掉。

### 3.2 建筑

`BuildingGenerator` 里 `wallBlock`/`floorBlock` 目前是在生成开始时解析一次的固定字符串（`resolveBlockId(footprint.getWallMaterial())`），这次要删掉这种"提前解析一次"的写法，改成在实际逐格放置墙体/楼板方块的循环内部，对每个位置调用 `MaterialMixResolver.resolve(footprint.getWallMaterial(), pos, footprint.getId())`（楼板同理，用 `footprint.getFloorMaterial()`）。

## 4. UI：材质选择器支持"添加点缀材质"

### 4.1 `BlockConfigNativeScreen` 新增调色盘选择入口

现有 `BlockConfigNativeScreen` 已经有 `SelectionMode.PALETTE` 这个模式（默认构造函数用的就是这个），但目前没有一个类似 `forSingleSelection` 那样"传入初始值、选完通过回调拿到结果"的便捷入口——现有的调色盘模式是给 `LineToBlockHandler` 那种"读取全局 `BlockConfigManager` 单例状态"的场景用的，不适合直接拿来给某个具体的材质字段（比如"这条路的点缀材质"）单独编辑，因为不同材质字段各自编辑的时候会互相覆盖同一个全局调色盘状态。

新增一个静态工厂方法：
```java
public static BlockConfigNativeScreen forPaletteSelection(
        Screen parent,
        List<String> initialBlockIds,   // 打开时预先把这些方块放进调色盘
        Consumer<List<String>> onConfirm);  // 关闭时把最终调色盘内容通过回调返回
```
实现思路：打开前调用 `BlockConfigManager.getInstance()` 的方法把 `initialBlockIds` 设置进调色盘（如果没有现成的"设置调色盘内容"方法，加一个，参考现有 `paletteBlocks.add(block)` 那部分逻辑，注意加之前要先清空，不要跟上一次别的地方留下的调色盘状态混在一起）；`close()` 方法里，在现有逻辑基础上，如果构造时传入了 `onConfirm` 回调，读取 `configManager.getSelectedBlockIds()` 并调用回调，把结果传出去（参考单选模式 `onBlockSelected.accept(...)` 的调用方式）。

这次任务只需要**最多两个材质**（主材质 + 点缀材质），不需要真的用到 14 个格子那么多，但复用同一个调色盘选择界面没问题，UI 上不需要限制成"只能选两个"——如果用户在这个界面里选了两个以上，取前两个作为主材质和点缀材质即可（第一个选中的当主材质，第二个当点缀，超过两个的忽略，弹窗关闭时可以给一个提示"最多支持两种材质，已保留前两种"）。

### 4.2 材质按钮 UI 调整（道路/建筑各自的编辑面板）

现有 `RoadUiWidgets.renderBlockMaterialPicker`（单选版本，弹出 `forSingleSelection`）需要新增一个用于 `MaterialMix` 的版本，或者在同一个方法里加分支，参考现有代码结构你自己判断怎么改动更小。UI 呈现：
```
路面材质：[方块图标+名称按钮，点击弹出调色盘选择]
  点缀比例：[========|--] 15%   （只有设置了点缀材质才显示这个滑条，没设置点缀材质时这行不显示）
```
点击材质按钮弹出 `forPaletteSelection`（用当前 `MaterialMix` 的 primary/accent 作为 `initialBlockIds`），关闭后把返回的列表映射回 `MaterialMix`（第一个 = primary，第二个如果有 = accent，只有一个就是 `accentMaterial = null`）。

`RoadDefaultParamsPanel`/`BuildingPlugin` 编辑 Tab 里原来单选材质按钮的地方，都要换成这个新版本，对应的三处（道路路面、建筑外墙、建筑楼板）都要改，其它材质按钮（人行道、路肩、地基、屋顶等）保持原来的单选版本不动。

## 验收标准

1. `MaterialMixResolver.resolve`：同一 `pos`/`seedKey` 多次调用结果一致；不同 `seedKey` 对同一 `pos` 的选择分布不同；固定 `mix` 和 `seedKey`，对大量不同 `pos` 统计点缀材质出现比例，应该接近 `accentRatio`（允许百分之几的误差）。
2. `hasAccent() == false` 时（没设置点缀材质），`resolve` 应该对任意 `pos` 都直接返回主材质，不应该有随机计算的开销（可以通过单测验证结果，不强制要求测性能）。
3. 旧存档兼容：构造一份 JSON，`carriagewayMaterial` 字段是纯字符串（模拟旧格式），验证 `RoadNetwork.fromJson` 能正确读取并转换成 `MaterialMix.single(...)`，不会解析失败或者报错。
4. 生成一段道路，`carriagewayMaterial` 设置了点缀比例 20%，验证生成结果里确实有大约 20% 的路面方块是点缀材质，且同一个位置（如果重新生成一遍）选出来的还是同一种材质。
5. 建筑外墙/楼板同理验证。
6. UI：点击材质按钮弹出的调色盘选择界面，选择两个方块后关闭，验证对应的 `MaterialMix` 正确更新（第一个是主材质、第二个是点缀材质）；只选一个时点缀材质应该被清空为 `null`。
7. 现有道路/建筑相关测试套件全部保持通过（尤其是涉及 `carriagewayMaterial`/`wallMaterial`/`floorMaterial` 字段类型变化后，确认没有编译错误或者遗漏的调用点）。

## 实施顺序

1. 第 1、2 章：`MaterialMix`/`MaterialMixResolver` 纯函数层，先写测试跑通，这一步完全不涉及 UI 和生成逻辑。
2. 第 4.1 节：`BlockConfigNativeScreen` 新增调色盘选择入口，这是独立的通用能力，先做好并验证能正常弹出、正常通过回调拿到结果。
3. 第 3 章接入道路生成逻辑，做完实机测试一次，确认路面材质混合效果符合预期。
4. 第 3 章接入建筑生成逻辑，同理测试。
5. 第 4.2 节 UI 接入两个插件的编辑面板，最后做。

每一步做完建议进游戏实际生成一段路/一栋楼看效果，材质混合的视觉效果好不好看，光靠代码审查和单元测试判断不出来，这是这次任务里最值得肉眼验证的地方。
